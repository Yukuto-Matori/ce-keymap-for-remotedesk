# アーキテクチャ仕様書

実装に関するアーキテクチャ仕様を定義

## 本番稼働ターゲット

- Windows11

## 開発環境

- Windows11, .NET Framework 4.8 SDK
- C# 言語
- WinForms

## 実行形態

- タスクトレイ常駐アプリ (WinForms + `NotifyIcon`)
- メインウィンドウは持たず、タスクトレイのコンテキストメニューから全操作を行う
- Mutex を用いた多重起動防止（二重起動時は新規プロセスを即終了し、既存インスタンスへフォーカス等は行わない）
- アプリアイコンは `resources/appicon_512x512.png` を使用する（タスクトレイアイコン・exeアイコン共通）

## 配布方針

- 配布形態は exe 単体のみとし、インストーラーは用意しない
- 配布 exe にはコード署名を行う

## 内部層構成

シンプルな4層構成とする。小規模アプリのため、責務分離は最小限に留める。

1. **UI層**
   - タスクトレイアイコン (`NotifyIcon`) とコンテキストメニュー
   - キーバインド変更ダイアログ、機能ON/OFF切り替えメニュー、自動起動設定メニュー
   - キーバインド変更はドロップダウン選択方式とする（キー押下キャプチャ方式は採用しない）
   - 修飾キーなし、またはメインキーのみ（修飾キー未選択）の組み合わせは許可しない（入力バリデーションで保存を拒否する）
2. **ホットキー検知層**
   - `SetWindowsHookEx` (`WH_KEYBOARD_LL`) によるグローバル低レベルキーボードフック
   - 押下中の修飾キー・メインキーの組み合わせを監視し、登録済みキーバインドと一致したら機能実行層へ通知
   - フック処理はできるだけ軽量にし、フックプロシージャ内で重い処理は行わない（機能実行は非同期にディスパッチする）
   - フックの解除（`UnhookWindowsHookEx`）に失敗した場合は、メッセージボックスでエラーを表示したうえでアプリを強制終了する
   - 機能間でキーバインドが重複した場合は「後勝ち」とする。後から設定された機能が該当バインドを取得し、以前そのバインドを保持していた機能はキーバインド「なし」の状態となり、自動的にOFFになる
3. **機能実行層**
   - 各ショートカット機能（アプリウィンドウ切り替え／拡大率変更 デスクトップ用／拡大率変更 モバイル用／Winキー押下）の実処理を担当
   - アプリウィンドウ切り替え: OS標準の Alt+Tab 相当の動作を `SendInput` でキー入力をエミュレートして実現
   - 拡大率変更: Windows の表示スケーリング(DPI設定)をレジストリ／API経由で変更し、即時反映を想定。反映に追加操作（ログオフ等）が必要な場合はアプリから促す、または実行する
   - Winキー押下: `SendInput` で `VK_LWIN`（左Windowsキー）の押下・解放をエミュレートし、Windowsキー単体を押したのと同じ動作（スタートメニュー表示等の副作用も許容する）を実現する。左右の切り替えは行わず、左Winキー固定とする
4. **設定永続化層**
   - `settings.json` の読み書き（アプリの実行フォルダ(exeと同じフォルダ)に配置）
   - PC起動時自動起動のためのレジストリ Startup キーの登録／解除
   - `settings.json` が破損している（パース不能）場合は、エラーダイアログを表示したうえでデフォルト設定を復元する
   - `settings.json` のバージョン (`Ver`) がアプリの想定バージョンと異なり、一部項目にデフォルト値を適用した場合は、ダイアログでユーザーに通知する

## ショートカット機能仕様

| 機能 | デフォルトキー | 処理内容 |
|---|---|---|
| アプリウィンドウ切り替え | 右Alt + 右Shift | `SendInput` で Alt+Tab 相当のキー入力をエミュレートする |
| 拡大率変更 デスクトップ用 | 右Alt + D | Windows表示スケーリング(DPI)を、設定で指定された「デスクトップ用」の拡大率に変更する |
| 拡大率変更 モバイル用 | 右Alt + M | Windows表示スケーリング(DPI)を、設定で指定された「モバイル用」の拡大率に変更する |
| Winキー押下 | 右Shift + W | `SendInput` で Windowsキー単体を押下したのと同じキー入力をエミュレートする |

- 各機能はキーバインド変更・個別ON/OFFが可能
- 拡大率変更機能は、拡大率の値自体も設定で指定可能
- キーバインドは修飾キー＋メインキーの組み合わせを必須とし、修飾キーなし／メインキーのみの登録は不可とする
- 機能間でキーバインドが重複した場合は後勝ち（新しく設定した機能がバインドを取得し、以前保持していた機能はキーバインド「なし」かつOFFになる）

## 設定ファイル (settings.json) 仕様

- 保存場所: アプリの実行ファイル(exe)と同じフォルダ
- 形式: 機能ごとにキーを持つオブジェクト構造（修飾キー＋メインキーの組み合わせを機能ごとに保持するシンプルな構造）

```json
{
  "Ver": 1,
  "AppWindowSwitch": {
    "Enabled": true,
    "Modifiers": ["RAlt", "RShift"],
    "Key": null
  },
  "ZoomDesktop": {
    "Enabled": true,
    "Modifiers": ["RAlt"],
    "Key": "D",
    "ZoomPercent": 100
  },
  "ZoomMobile": {
    "Enabled": true,
    "Modifiers": ["RAlt"],
    "Key": "M",
    "ZoomPercent": 150
  },
  "PressWinKey": {
    "Enabled": true,
    "Modifiers": ["RShift"],
    "Key": "W"
  },
  "AutoStart": false
}
```

- 他PCへのファイルコピーによる設定共有を想定するため、環境依存のパス等は保持しない
- `Ver` フィールドで設定ファイルのスキーマバージョンを管理する
- アプリ更新時の後方互換性:
  - 新バージョンで項目が追加された場合、旧バージョンの `settings.json` に存在しない項目はデフォルト値を使用する
  - 読み込んだ `settings.json` の `Ver` がアプリの想定 `Ver` と異なり、一部項目にデフォルト値を適用した場合は、その旨をダイアログでユーザーに通知する
  - 読み込み・書き込み後、`settings.json` の `Ver` はアプリの想定バージョンに更新する
- ファイルが破損している（JSONとしてパース不能）場合は、エラーダイアログを表示したうえで全項目をデフォルト値で復元する

## ロギング・エラーハンドリング方針

- アプリの実行ファイル(exe)と同じフォルダに `log.txt` 等のログファイルを出力するファイルログを導入する
- 想定外の例外は、ログファイルへの記録に加えて必要に応じてメッセージボックス等でユーザーに通知する
- 過度に複雑なロギング基盤（外部ライブラリによる高度な構造化ログ等）は導入せず、シンプルなファイル追記方式とする

## テスト方針

- 自動テストは最小限とし、動作確認は手動（実機での動作確認）を中心とする
- 単体テストの対象は、設定ファイルのパース処理など、OS依存のない純粋ロジックに限定する
- キーボードフックや `SendInput`、DPI変更などOSに依存する処理は自動テスト対象外とする
